# Submission Notes

## Setup for Reviewers

1. Clone the repo and copy `.env.example` to `.env`
2. Fill in your own `GITHUB_OWNER`, `GITHUB_REPO`, and `GITHUB_TOKEN`
3. Run `npm install && npm run install:browsers`
4. Run `node saveSession.js` â€” a browser will open, log in with your GitHub account, the session will be saved automatically to `.auth/github-storage-state.json`
5. Run `npx playwright test tests/github/` to execute all tests

## Assumptions

- GitHub's browser UI communicates exclusively through GraphQL (`/_graphql`), not the REST API. All `waitForResponse` filters target this endpoint since no REST call is made from the browser during UI interactions.
- Authentication is handled via a `storageState` file generated by `saveSession.js`. The script opens a browser, lets the user log in manually to GitHub, then saves cookies and local storage to `.auth/github-storage-state.json`. This file is reused across all tests via Playwright's `storageState` config, avoiding login in every test. The session is assumed to remain valid for the full test run â€” no automatic refresh or re-login mechanism was implemented.
- Each test creates its own issue to guarantee full isolation. No test depends on state from another test.

## Tradeoffs

- **`expect.poll` over raw assertions**: After each UI action, I use `expect.poll` to retry the API check until the backend state is consistent. This avoids `waitForTimeout` entirely while handling GitHub's eventual consistency gracefully.
- **`Promise.all` + `waitForResponse` on `/_graphql`**: The browser never hits `/issues` or `/comments` REST endpoints â€” everything goes through GraphQL. Filtering on `/_graphql` with status 200 is the most reliable way to synchronize network-triggering actions without hardcoding internal GitHub endpoints.
- **Cleanup in `finally` with `.catch(() => {})`**: If a test fails and the request context is disposed, the cleanup call would throw and mask the original error. The `.catch` prevents this while still attempting cleanup on success.
- **No Page Object Model**: As instructed. Locators live directly in the test, sourced from `page.scanDOM()` and Codegen.

## What I Would Improve With Another Hour

- Add a retry wrapper around `_addIssueComment` to handle GitHub's propagation delay on freshly created issues (occasional 422 errors).
- Extract a shared constant for the `waitForResponse` GraphQL filter to reduce duplication across spec files.
- Add a global teardown that closes any orphaned issues left by failed test runs.
- Investigate filtering `waitForResponse` by GraphQL mutation name in the request body (e.g., `closeIssueMutation`) for more precise synchronization.
